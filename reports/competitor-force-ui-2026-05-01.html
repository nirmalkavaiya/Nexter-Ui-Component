Verdict: Nexter UI wins on breadth & portability — Force UI wins on tooling maturity
Nexter UI ships 52 components (vs 43), has zero runtime dependencies (Force UI bundles framer-motion, styled-components, recharts, and 12 more), supports React 17+ (broader WP compatibility), and uses an MIT license with no private registry lock-in. However, Force UI has a decisive advantage in developer experience: full TypeScript with .d.ts, Storybook 10 with a11y + vitest addons, Playwright/Chromatic visual regression CI, and automated axe accessibility testing. The gap to close is infrastructure, not components — adding TypeScript and Storybook would make Nexter UI the clear winner.

🎯 SWOT — Nexter UI vs Force UI
Evidence-based analysis from repository signals.
💪 Strengths (Nexter)
  • 52 components vs 43 — largest coverage in the niche
  • Zero runtime deps — lightest possible consumer bundle
  • React 17+ support — works in all WP admin environments
  • MIT license — commercially unrestricted
  • Public install — no auth token or private registry needed
  • No Tailwind requirement in consumer project
  • WordPress-specific components (FeatureToggleCard, StatCard, CopyInput, OTPInput)
  • Clean CSS token system — easy brand theming
  • 0 open issues — stable baseline
  • Actively maintained (last commit: yesterday)
⚠ Weaknesses (Nexter)
  • No TypeScript — no props autocomplete, no type-safety
  • No Storybook — no interactive component docs
  • No unit or E2E tests — regressions are invisible
  • No a11y tooling — WCAG coverage is unverified
  • No visual regression CI — UI drift goes undetected
  • No chart components — analytics dashboards need external lib
  • GitHub repo has no description or topics — poor discoverability
  • npm publish status unverified — install claim may fail
  • No ESLint / Prettier setup — inconsistent code quality
🚀 Opportunities
  • Add TypeScript — immediately close Force UI's biggest DX advantage
  • Add Storybook — create the interactive docs Force UI already has
  • Publish to npm public registry — enable one-command install
  • Add Drawer + Sidebar — cover the two Force UI-unique layout components
  • Add ActivityFeed + CommandPalette — no competitor has these for WP admin
  • Add Recharts-backed ChartCard — match Force UI analytics capability
  • Document "zero deps" as a flagship differentiator in README hero
  • Add GitHub topics + description — improve organic discoverability
  • WP media integration in FileUpload is unique — market it
⚡ Threats
  • Force UI backed by Brainstorm Force (Astra, CartFlows) — larger team
  • If Force UI goes public registry — adoption barrier drops dramatically
  • TypeScript adoption gap widens if not addressed — devs increasingly expect types
  • Force UI's Storybook + Chromatic setup makes it easier for contributors
  • 53 open issues on Force UI = active community — could accelerate their roadmap
  • React 19 could widen compatibility story if Force UI doesn't update

🗺️ Recommended Roadmap — Close the Gap in 3 Sprints
Ordered by impact × effort. Fix the critical gaps first, then extend unique advantages.
1
Add TypeScript + .d.ts declarations
Convert all JSX to TSX, add tsconfig.json, wire vite-plugin-dts. This alone closes Force UI's biggest DX advantage. Target: all 52 components typed within 2 sprints. Start with the highest-use ones (Button, Input, Modal, Toggle).
High effort · Critical
2
Publish to npm public registry
Run npm publish --access public and verify npm install nexter-ui-component works. Add publishConfig to package.json. This makes the install story in the README actually true and removes the GitHub URL workaround.
Low effort · High impact
3
Add GitHub repo description + topics
Set: Description = "40+ accessible React UI components for WordPress admin dashboards — zero dependencies, MIT license". Topics: react, ui-components, design-system, wordpress, admin-ui, dashboard, component-library. Takes 5 minutes, boosts discoverability significantly.
15 min · Quick win
4
Add sideEffects: false to package.json
This single package.json field tells bundlers the library is fully tree-shakeable. Force UI already has it. Without it, bundlers may include unused components in consumer builds. Add "sideEffects": ["**/*.css"] to preserve the style import.
30 min · Important
5
Add Storybook + basic stories for all components
Install @storybook/react-vite. The stories already exist (MultiSelect.stories.jsx, FeatureToggleCard.stories.jsx) — wire them up. Add @storybook/addon-a11y from day one to start catching accessibility issues automatically. Force UI uses Storybook 10 — match that.
Med effort · High DX value
6
Add Vitest tests (start with 10 components)
Prioritise: Button, Input, Toggle, Modal, Dropdown, Select, Checkbox, DatePicker, OTPInput, ColorPicker. Use @testing-library/react. Force UI has full coverage — even partial test coverage is a trust signal for enterprise consumers.
Med effort · Trust signal
7
Add Drawer + Sidebar components
The two Force UI-unique layout components most used in dashboard shells. Drawer = Sheet with directional animation (already have Sheet, extend it). Sidebar = vertical nav with collapse, active state, nested groups. Sheet component provides a head-start for Drawer.
Med effort · Closes gap
8
Add ChartCard components (using Recharts)
Force UI ships 4 chart types. Add a lightweight ChartCard wrapper around Recharts (as an optional peer dep, not bundled) for Line, Bar, Pie, Area charts pre-styled with nxp tokens. Keep Recharts as peerDep so it doesn't bloat consumers who don't need charts.
Med effort · Parity
9
Market "Zero Dependencies" as hero differentiator
Add a bundle size comparison to README.md: "Force UI bundles 15 runtime deps including framer-motion (~50KB), styled-components, and recharts. Nexter UI ships zero. Your users pay nothing extra." Add a bundle size badge. This is a defensible, measurable, permanent advantage.
Low effort · Marketing
